🎮 Godot 工具集插件 — 产品设计文档

项目: Godot Adaption Kitset for BoBanana 5.0 | 版本: v0.1-DRAFT | 类型: 非技术产品设计

📌 一句话定位
将 Godot 引擎变成 AI Agent 的"游戏沙盒"——Agent 通过结构化工具操控引擎，用户用自然语言做游戏。

1. 用户访谈提纲

目标用户

  • P1 — 独立游戏开发者（全职/副业）：会用 Godot/Unity，想用 AI 加速游戏开发
  • P2 — 游戏开发初学者：有编程基础但没做过完整游戏，希望 AI 带着做
  • P3 — AI 工具爱好者：已经在用 Cursor/Copilot，想试 AI + Godot 组合

招募条件

  • 用过 Godot 4.x（至少打开过编辑器）
  • 听说过或试过 AI 辅助编程（Copilot/Cursor/Claude）
  • 最近 3 个月有游戏开发尝试

访谈问题（8 个）

  1. 你目前做游戏开发，最耗时的是哪个环节？
    追问：编码？场景搭建？资源管理？调试？为什么最耗时间？
  2. 你试过用 AI 辅助做游戏吗？用的什么工具？效果怎么样？
    追问：有没有遇到"AI 不懂游戏引擎"的感觉？比如生成的代码跑不了？
  3. 如果有一款工具能让 AI Agent 直接操作 Godot 编辑器——创建场景、挂脚本、调参数——你觉得有用吗？每周你会用几次？
    封闭：每天/每周几次/每月/不会用
  4. 你愿意在 AI 辅助工具上花多少钱？
    封闭：$0（只用免费的）/ $5-10/月 / $10-20/月 / $20+/月
  5. 你用过 Godot 的插件系统吗？（addons 目录下的 EditorPlugin）
    追问：自己写过插件吗？期望插件能提供什么功能？
  6. 如果有 AI 能看懂你的需求文档并自动搭建 Godot 项目结构和场景，你会信任它吗？还是每一步都要人工确认？
    追问：什么场景下你敢让 AI 全自动？什么场景必须每一步审查？
  7. 你是从哪个引擎转到 Godot 的？（还是直接上的 Godot？）
    追问：如果 Unity 的 AI 工具更好，你会回去吗？Godot 吸引你的是什么？
  8. 如果这个工具开源免费但需要自己配 API Key，和一个 $15/月开箱即用，你选哪个？
    封闭：开源自配 / $15/月省事 / 都不选（说明原因）

3. 假设验证卡片

#假设验证方法成功标准时间责任人
H1 🔴 独立开发者愿意用 AI Agent 操控 Godot 编辑器来加速开发 MVP 内测 + NPS 问卷（招募 20 人，每人用 2 周） NPS ≥ 30，≥ 60% 每周使用 ≥ 3 次 4 周 产品+开发
H2 🟡 Godot 插件（WebSocket）比 CLI 桥接更受 Agent 欢迎（开发效率更高） A/B 测试：同一批任务，Plugin 组 vs CLI 组 Plugin 组完成任务速度快 30%，用户满意度高 20% 3 周 开发
H3 🟢 用户愿意为 AI Godot 工具付费 ≥ $9/月 定价页面 A/B 测试 + 预注册转化率 预注册转化率 ≥ 3%（从 landing page 到注册） 2 周 产品+市场
🔍 悲观预判
H1 最可能失败：用户口头说"想要"，实际 2 周后只剩 30% 活跃。NPS 可能被稳定性问题拉低到 0-10。建议提前准备"快速失败"标准：2 周内活跃率 < 40% 则 pivot。

阶段用户行为想法痛点机会点

🚀		 安装插件 → 配置 API Key → 试创建项目 "能成功吗？"→🤩"居然一次过了！" 或 😤"哪里报错了？" AI API Key 配置麻烦；Godot 路径检测失败；首次成功率可能低于 50% install.bat 一键安装 + 自动检测 Godot 路径；首次体验做成引导式的
🔄 持续使用
🔄		 日常用 AI 写脚本、搭场景、调试游戏 "写 GDScript 快多了" / "怎么又偷偷改了老逻辑？" AI 不理解已有代码逻辑；GDScript 训练数据少导致生成质量不稳 提供 "范围限定" 机制（只改指定文件）+ GDScript 训练数据集微调

8. 需求评审清单

#自查问题为什么重要
1用户真的需要这个吗？还是我们觉得他们需要？如果用户不觉得"AI 操控引擎"比"AI 写代码"更实用，就会用回 Copilot
2MVP 能 2 周内交付吗？Godot 的 AI 生态在快速变化（gda/Fennara/Ziva 都在迭代），慢了就被竞争对手覆盖
3没有 Godot 插件能工作吗？（CLI fallback 是否足够？）插件是差异化核心但不是 MVP 必要条件。CLI 桥接足够让用户先用起来
4中文用户能否在 5 分钟内成功跑起来？中国用户需要国内可访问的 API、GitHub 不墙、中文 README。任何一步卡住就放弃
5AI 生成错误代码时用户怎么办？必须有"回滚"机制和"手动修复引导"。否则一次生成错误就永久失去信任
6我为什么选 Godot 而不是 Unity/Unreal？Godot 选择既是机会（蓝海）也是风险（用户少）。定位必须足够清晰
7长期维护谁来做？需要跟着 Godot 版本升级（4.x 到 5.0）。否则会像某些 Godot 插件一样版本更新后废弃
8社区是否在做一个比我更好的方案？gda 的 stars 在快速增长。如果在 1 个月内被反超，说明需求已被满足，不做
🔍 悲观预判
Q4（中文用户 5 分钟启动）最难做到。GitHub 在中国部分地区访问不稳定，Node.js 安装、API Key 配置、Godot 路径检测任何一个环节都可能让用户放弃。建议准备一个完全离线的"体验版"（内置示例项目，不需要 API Key）。
