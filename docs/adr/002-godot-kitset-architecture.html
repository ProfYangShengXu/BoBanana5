ADR-002: Godot Kitset 架构决策

✅ 接受 | 2026-07-19

背景

需要将 Godot-adaption-kitset 从独立仓库迁移到 BoBanana 5.0 的 Kitset 体系，并新增游戏工作室角色卡和 Godot 通信插件。

C1: 迁移方式 — 复制 vs 子模块

决定: 全量复制 + 独立版本

对比方案：

  • A — 全量复制 ✅：将 Godot-adaption-kitset 的工具复制到 kitsets/godot/tools/godot-tools/。优点：完全解耦、无外部依赖。缺点：需要手动同步更新。
  • B — Git 子模块：引用原始仓库。优点：自动同步。缺点：增加 Git 复杂度、依赖外部仓库可用性。
  • 选用 A：因为 Bobanana 的 kitset 体系要求自包含，且 godot 工具集还在快速迭代，解耦更灵活。

C2: 通信协议 — WebSocket vs stdio vs TCP

决定: WebSocket JSON-RPC 2.0

  • WebSocket：Godot 原生支持（WebSocketPeer），全双工，双向实时。端口 9800。
  • stdio：不适合外部进程→Godot 场景（Godot 不是子进程）。
  • TCP：可以选择但 WebSocket 更易用、更标准。

C3: 角色卡生成策略

决定: 快速引用模式 + 统一模板

25 张游戏工作室角色卡使用 HR 技能的"快速引用模式"生成，跳过研究步骤直接产出角色卡。每角色仅产出 role-card.yaml + SKILL.md + standards-brief.yaml 三个文件。game-architect 手工编写（OP 角色，格式对齐 quant-architect）。
