# docs — 产品设计文档

**语言指令：推理用英文，回复用中文。**

## 核心原则

```
本 skill 只回答 WHAT 和 WHY，不回答 HOW。
不写一行代码、不画一张架构图、不定一项技术栈。
HOW 由 pipeline 架构师负责。
每项结论必须有外部数据源支撑，不准凭空编造。
```

## 产出物

1. `docs/product/<项目名>-product-prd.html` — 产品设计文档（浏览器打开，给人读的）
2. `docs/product/<项目名>-product-prd.yaml` — 结构化产品定义（机器可读，pipeline 架构师直接引用）

两个文件必须同时产出，内容一致。每项数据必须在文档中标注来源 URL。

---

## 工作流程

```
第 1 步：解析目标 → 确定研究维度
第 2 步：并行派发研究子 Agent → 收集真实数据
第 3 步：基于研究结果合成 HTML + YAML
第 4 步：标注数据来源，输出最终文档
```

---

## 第 1 步：解析目标

从用户输入中提取：
- 产品名称、一句话描述
- 目标用户群体
- 所属行业/领域
- 核心功能方向

如果信息不足，用 `web_fetch` 搜索行业相关背景补充理解。

## 第 2 步：并行研究

根据目标确定需要的研究维度。**至少并行派发 2 个研究子 Agent**，可用维度如下：

### 研究维度表

| # | 维度 | 适用场景 | 必做？ |
|---|------|---------|-------|
| 1 | **竞争格局** | 大多数 B2C/B2B 产品 | ✅ 推荐 |
| 2 | **用户需求** | 面向大众用户的产品 | ✅ 推荐 |
| 3 | **市场趋势** | 新兴领域/技术驱动产品 | ⚠️ 按需 |
| 4 | **技术/功能参考** | 有具体功能列表的产品 | ⚠️ 按需 |

如果目标涉及的行业明确（如"AI 笔记"、"在线教育"），至少做竞争格局 + 用户需求两个维度。

---

### 子 Agent 模板

#### 🔍 研究子 Agent：竞争格局

用 `task` 派发，description 建议 `"竞争分析：<项目名>"`：

```
你是一个产品竞争分析研究员。你的任务是通过全网搜索，为以下产品提供竞争格局分析。

产品目标：<用户原始目标>

请使用 web_fetch 搜索以下信息，每个数据点标注来源 URL：

1. **直接竞品**：至少找到 5 个直接或间接竞品
   - 搜索关键词：<行业> + 竞品 / alternatives / competitors / 对比
   - 数据源：ProductHunt、G2、知乎/小红书对比帖、36氪、行业报告
   - 每个竞品输出：名称、一句话定位、目标用户、核心功能、定价模式、优缺点

2. **竞争格局总结**：
   - 市场集中度（头部玩家/长尾市场）
   - 竞品之间的差异维度（价格/功能/体验/生态）
   - 空白机会点（竞品没做好或没做的）

3. **价格参考**：同类产品的定价区间

输出格式为 Markdown，每个竞品用 ### 标题分隔，末尾附所有来源 URL 列表。
```

#### 🔍 研究子 Agent：用户需求

用 `task` 派发，description 建议 `"用户研究：<项目名>"`：

```
你是一个用户研究分析师。你的任务是通过分析公开信息，为以下产品构建真实的用户画像和需求分析。

产品目标：<用户原始目标>

请使用 web_fetch 搜索以下信息，每个数据点标注来源 URL：

1. **目标用户是谁**？
   - 搜索同类产品的用户评价/评论/论坛讨论
   - 数据源：知乎、V2EX、Reddit、小红书、应用商店评论、G2 用户评价
   - 提取：用户身份、使用场景、具体痛点、期望功能、替代方案使用体验

2. **用户画像**：构建 3-5 个真实感画像
   - 每个画像有：身份、场景、痛点、频率、技术熟练度、现用替代方案
   - 不准用"普通用户/高级用户"——必须具体到职业和场景

3. **需求优先级**：
   - Must-have（不做就没法用的功能）
   - Performance（慢了会流失的性能要求）
   - Delight（超出预期的体验点）

输出格式为 Markdown，每个画像用 ### 标题分隔，末尾附所有来源 URL 列表。
```

#### 🔍 研究子 Agent：市场趋势

用 `task` 派发，description 建议 `"市场趋势：<行业名>"`：

```
你是一个市场研究分析师。你的任务是分析以下产品所在行业的市场趋势。

行业/领域：<从目标中提取>

请使用 web_fetch 搜索以下信息，每个数据点标注来源 URL：

1. **市场规模**：行业规模（亿元/亿美元）、增长率（CAGR）、历史数据
   - 数据源：行业报告（艾瑞/IDC/Gartner/Statista）、券商研报、新闻稿

2. **发展趋势**：
   - 技术趋势（AI/云/移动/...）
   - 政策趋势（监管/扶持/标准）
   - 用户行为变化

3. **竞争动态**：
   - 近期融资事件（哪些公司拿了多少钱）
   - 新进入者（跨界竞争者）
   - 并购/退出

4. **风险因素**：
   - 市场风险（天花板/饱和/萎缩）
   - 政策风险
   - 技术替代风险

输出格式为 Markdown，末尾附所有来源 URL 列表。
```

#### 🔍 研究子 Agent：技术/功能参考

用 `task` 派发，description 建议 `"功能参考：<项目名>"`：

```
你是一个产品功能研究员。你的任务是为以下产品寻找同类功能的最佳实践和参考。

产品功能方向：<从目标中提取>

请使用 web_fetch 搜索以下信息，每个数据点标注来源 URL：

1. **同类产品功能对比**：
   - 搜索同类产品的功能列表、定价页面、产品文档
   - 提取功能矩阵（A产品有 X、B产品有 Y）

2. **行业标准做法**：
   - 这类产品通常包含哪些模块？
   - 常见的技术选型是什么？
   - 有哪些已知的设计模式和坑？

3. **创新功能参考**：
   - 最近出现的差异化功能
   - 用户呼声高但竞品还没做的功能

输出格式为 Markdown，末尾附所有来源 URL 列表。
```

### 派发规则

```python
# 伪代码逻辑——据此决定派发哪些子 Agent

dimensions = []

# 竞争格局：大多数产品都需要
dimensions.append("competitor")

# 用户需求：面向大众用户时必做
if 目标用户不是纯企业内部工具:
    dimensions.append("user_research")

# 市场趋势：新兴领域或技术驱动型
if 领域是"AI / Web3 / 新能源 / 生物科技 / ...":
    dimensions.append("market_trend")

# 技术参考：有明确功能列表时
if 目标包含具体功能描述:
    dimensions.append("tech_reference")

# 至少 2 个，至多 4 个并行
for dim in dimensions:
    task(description="研究：" + dim, prompt=研究子Agent模板)
```

**并行派发**：用 `task` 同时派发所有研究子 Agent，等待全部返回后再进入第 3 步。不准串行派发。

---

## 第 3 步：合成文档

收集所有子 Agent 的研究 Markdown 报告后，基于真实数据合成 HTML + YAML。

### 合成规则

| 文档章节 | 数据来源 | 填充方式 |
|---------|---------|---------|
| 1. 背景与目标 | 原始目标 + 研究子 Agent 的背景理解 | 总结行业背景，引用市场数据 |
| 2. OKR 指标 | 市场趋势子 Agent | 基于行业基准设定可量化的目标 |
| 3. 用户画像 | 用户研究子 Agent | 直接移植研究结果的画像 |
| 4. MVP 定义 | 所有子 Agent 的综合 | 基于竞品功能矩阵 + 用户需求排优先级 |
| 5. 2B/2C 分析 | 竞争格局子 Agent | 分析目标客群的付费模式和获客方式 |
| 6. 竞争分析+SWOT | 竞争格局子 Agent | 直接移植竞品矩阵 + SWOT |
| 7. 设计理念 | 主控判断（不依赖研究） | 基于产品定位描述体验方向 |
| 8. 风险 | 市场趋势 + 竞争子 Agent | 基于真实行业数据评估风险 |

### 数据来源标注

**每个表格和关键数据点后面**，必须用上标标注来源：

```
| 竞品 | 定位 |
|------|------|
| Notion | All-in-one workspace [¹] |
| Obsidian | Local-first knowledge base [²] |

---
[¹] https://www.notion.so
[²] https://obsidian.md
```

如果子 Agent 研究结果中某个数据点缺乏明确的 URL 来源，在文档中标记为 `[来源待确认]`，不准编造 URL。

## 🎨 HTML 样式规范

产出的 HTML 必须内嵌 CSS，使用以下风格：

```
- 深色标题栏（#1a1a2e 或类似深色）+ 浅色内容区（#f8f9fa）
- 响应式布局：桌面优先，移动端（≤768px）可读，使用 @media
- 可折叠章节：用 <details><summary> 包裹每个大节
- 粘性目录导航（sticky sidebar）：左侧或顶部，点击跳转锚点
- 打印友好：@media print 隐藏导航、折叠展开、缩小边距
- 表格：交替行颜色、左对齐表头
- 代码风格：项目名、文件名用等宽字体
- 顶部显示生成时间、版本标记、**数据来源总数**
- 末尾附录列出所有引用来源 URL（去重、按章节分组）
```

## HTML 必须包含

### 1. 背景与目标

- 这个产品解决什么**现实问题**？
- **一句话定位**是什么？（像 Airbnb for X / 类 Notion but for Y）
- 现存方案有什么痛点？（引用用户研究和竞品数据）
- 行业背景数据（引用市场趋势研究）

### 2. 数据指标（OKR 体系）

定义可量化的成功标准，**基于行业基准**：

| 层级 | 指标 | 目标值 | 衡量方式 | 行业基准 |
|------|------|-------|---------|---------|
| North Star | 北极星指标 | 例如"周活跃项目数 ≥ 1000" | 埋点/日志 | 头部竞品 ~5000 |
| KR-1 | 关键结果 1 | 量化的里程碑 | 具体统计方式 | |
| KR-2 | 关键结果 2 | 量化的里程碑 | 具体统计方式 | |
| KR-3 | 关键结果 3 | 量化的里程碑 | 具体统计方式 | |

每个 KR 必须可量化、有时间限制。行业基准来自研究子 Agent 的数据。

### 3. 用户画像（至少 3 类）

每类用户使用**固定表格模板**，数据来自用户研究子 Agent：

| 维度 | 画像 A：<名称> | 画像 B：<名称> | 画像 C：<名称> |
|------|---------------|---------------|---------------|
| 身份/场景 | | | |
| 核心痛点 | | | |
| 使用频率 | | | |
| 核心诉求 | | | |
| 技术熟练度 | | | |
| 替代方案 | | | |

每个痛点标注来源（用户评价/论坛帖子的 URL）。

### 4. MVP 定义

- MVP 必须包含的功能（must-have），按优先级排序——基于竞品功能矩阵 [来源]
- 一期后加的功能（nice-to-have）
- **明确不比什么**——说出来不做什么比做什么更重要
- MVP 的判断依据：竞品都有 + 用户呼声高 → must-have；竞品有但用户没提 → nice-to-have

### 5. 2B / 2C 分析

| 维度 | To C | To B |
|------|------|------|
| 用户是谁 | | |
| 付费意愿 | | |
| 决策链 | | |
| 核心功能差异 | | |
| 获客方式 | | |

定价参考来自竞品子 Agent。

### 6. 竞争分析与 SWOT

#### 竞争格局矩阵

| 竞品 | 定位 | 目标用户 | 定价 | 优势 | 劣势 | 与本产品差异 |
|------|------|---------|------|------|------|------------|
| 竞品 A | | | | | | |
| 竞品 B | | | | | | |
| 竞品 C | | | | | | |
| 竞品 D | | | | | | |
| 竞品 E | | | | | | |

至少列出 5 个竞品或替代方案（包含非数字替代）。数据来自竞争格局研究子 Agent。每个竞品标注信息来源 URL。

#### SWOT 分析

| | 正向 | 负向 |
|--|------|------|
| **内部** | **S（优势）**：基于研究数据提炼 | **W（劣势）**：基于研究数据提炼 |
| **外部** | **O（机会）**：来自市场趋势研究 | **T（威胁）**：来自竞争格局研究 |

### 7. 设计理念

- 体验关键词（"极简"、"专业"、"有趣"、"可靠"）
- 核心交互原则（不超过 3 条）
- 视觉风格方向（文字描述，不需要设计稿）
- 参考同类产品的设计风格 [来源]

### 8. 风险（非技术）

| 风险类别 | 风险描述 | 可能性 | 影响 | 缓解措施 | 数据来源 |
|---------|---------|-------|------|---------|---------|
| 市场风险 | 竞品抢先发布 / 需求不存在 | 高/中/低 | 高/中/低 | | 竞争格局研究 |
| 运营风险 | 内容审核 / 法规合规 | 高/中/低 | 高/中/低 | | 市场趋势研究 |
| 增长风险 | 冷启动困难 / CAC 过高 | 高/中/低 | 高/中/低 | | 竞品定价分析 |
| 变现风险 | 付费意愿不足 / 定价失误 | 高/中/低 | 高/中/低 | | 用户研究 |

## YAML 副产物格式

`docs/product/<项目名>-product-prd.yaml` 必须包含以下字段，**每个对象增加 sources 字段**：

```yaml
product:
  name: "<项目名>"
  one_liner: "<一句话定位>"
  created_at: "<生成时间 ISO-8601>"

sources:
  - url: "<来源 URL>"
    section: "<所属章节>"
    accessed_at: "<访问时间>"

success_metrics:
  north_star: "<北极星指标>"
  key_results:
    - name: "KR-1"
      description: "<描述>"
      target: "<量化目标>"
      measure: "<衡量方式>"

personas:
  - name: "<画像名称>"
    scenario: "<身份/场景>"
    pain_point: "<核心痛点>"
    frequency: "<使用频率>"
    need: "<核心诉求>"
    tech_level: "<技术熟练度>"
    sources: ["<来源URL1>", "<来源URL2>"]

competitors:
  - name: "<竞品名称>"
    positioning: "<定位>"
    pricing: "<定价>"
    strength: "<优势>"
    weakness: "<劣势>"
    difference: "<与本产品差异>"
    source: "<来源URL>"
```

HTML 为主（给人读），YAML 为辅（给架构师用）。YAML 中的 `sources` 字段是新增的，记录每条数据的来源。

## 完成

1. 派发并收集所有研究子 Agent 结果
2. `write_file` 写出 HTML（含引用来源附录）
3. `write_file` 写出 YAML（含 sources 字段）
4. 告知用户两个文件路径和数据来源数量
5. 注：后续在 pipeline 中架构师角色会读 prd.yaml，自动转成技术方案
