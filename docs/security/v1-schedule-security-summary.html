安全分析结论

✅ 最终安全评级：LOW（低风险） — 未发现可被利用的安全漏洞。所有攻击向量均有有效防护。

威胁模型分析

🔴 1. 命令注入（Command Injection）

攻击向量：通过 MCP handler 传递恶意 goal 参数，企图执行任意系统命令。

分析：

  • trigger_task() 第 303 行：subprocess.run([sys.executable, '-m', 'pipeline_orchestrator', 'init', goal], ...) — 使用 list 参数而非 shell 字符串，Python 的 subprocess.run 不经过 shell 解析。即使 goal 包含 ; rm -rf / 也只作为单个参数传递给 Python 进程。
  • _install_winsch_task() 第 153 行：action 字符串拼接 goal，但 goal 已在创建时经过 validate_goal() 白名单校验。
  • cmd_run() 第 57 行：同样使用 list 参数调用 subprocess。
✅ 防护：subprocess list 模式 + goal 输入校验白名单。风险等级：LOW

🔴 2. 路径穿越（Path Traversal）

攻击向量：通过 MCP handler 传递恶意 task_id，访问日志目录外的文件。

分析：

  • 所有接受 task_id 的函数（remove_task, trigger_task, get_task_logs）均先调用 validate_task_id()
  • validate_task_id() 使用正则 ^[a-f0-9]{8}$ — 仅允许 8 位 16 进制字符
  • 路径穿越尝试如 ../../.. 或 /etc/passwd 会被立即拒绝
✅ 防护：严格 task_id 校验 regex。风险等级：LOW

🟡 3. 输入验证绕过（Input Validation Bypass）

攻击向量：绕过 validate_goal 的字符白名单。

_GOAL_PATTERN = re.compile(r'^[\u4e00-\u9fff\w\s,.\-:;!?，。：；！？、（）()【】\[\]/@#%&+=~·]+$')

分析：

  • 白名单允许 /, ., @, #, %, &, + 等字符。
  • 但这些字符在 subprocess list 模式下不构成命令注入风险。
  • 白名单已过滤 <>（HTML/XML 标签）、'（单引号）、"（双引号）、$（变量替换）
  • 最大长度 200 字符防止缓冲区/日志滥用。
✅ 防护：白名单 + 长度限制 + subprocess list 模式三重防护。风险等级：LOW

🟡 4. MCP Handler 越权

攻击向量：MCP handler 无鉴权，可能被任意 Reasonix 会话调用。

分析：

  • MCP server 运行在 Reasonix 桌面端的 stdio 通道内，受 Reasonix 安全沙箱保护。
  • 所有 MCP handler 仅操作本地文件（tasks.json + 日志），不发起网络请求。
  • 操作范围限定在 pipeline 调度域（创建/列出/删除/执行/日志），不能执行任意系统命令。
  • schedule_create 的参数长度和字符集受限。
⚠️ 注意：Reasonix 的 trusted_write_tools 和 tool_timeout_seconds 配置可进一步限制 MCP handler 的行为。建议在 reasonix.toml 中为 schedule_* tools 设置独立的超时和权限。当前无需立即修复。

🟢 5. 日志安全

攻击向量：日志文件包含敏感信息或可被污染。

分析：

  • 日志文件仅存储 task_id、goal、rounds、时间戳、pipeline初始化输出摘要 —— 不包含密码/密钥/Token
  • 日志文件命名：{task_id}_{timestamp}.json — task_id 经过 hex 校验
  • 日志写入失败使用 try/except 捕获，不会崩溃
  • 多任务同时写入不存在竞争条件（每个日志文件独立）
✅ 合规：满足最小日志原则。风险等级：LOW
