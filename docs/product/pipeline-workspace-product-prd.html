📦 管线工作区隔离 — 产品 PRD

版本: v1.0 | 状态: 需求已确认 | 类型: 架构重构

1. 一句话定位

将管线状态从全局单例迁移到工作区隔离存储，让同一项目能同时运行多条独立管线。

2. 目标用户

  • 主要: Bobanana 5.0 开发者 — 需要同时运行 Quant Kitset 管线 + 主开发管线
  • 次要: 维护者 — 需要调试/修复管线状态不一致问题

3. 当前架构问题

问题后果严重性
.reasonix/state/machine_state.json 是全局单例无法同时运行两条管线🔴 高
has_pending_pipeline() 只扫单一目录并行管线互相误判为"进行中"🔴 高
cycle-bridge MCP 工具读固定路径多 session 交叉写入破坏状态🟡 中
状态文件与管线元文件分离存储repair 逻辑复杂，易不一致🟡 中

4. 方案

4.1 目录结构

.reasonix/pipelines/
├── {workspace}/
│   ├── {pipeline_id}/
│   │   ├── pipeline.json       ← 管线元数据（替代旧 .reasonix/pipelines/pl-*.json）
│   │   ├── machine_state.json  ← 状态机运行态（替代旧 .reasonix/state/）
│   │   ├── cycle_state.json    ← Cycle 工作流（替代旧 .reasonix/cycle/）
│   │   └── next_prompt.txt     ← 下一提示（替代旧 .reasonix/cycle/）
│   └── ... (更多管线)
└── .default/                   ← 向后兼容：未指定 workspace 时使用

4.2 核心改动

文件改动
pipeline_orchestrator.pyPIPELINE_DIR 改为 workspace 感知；init_pipeline/advance_pipeline/list_pipelines 增加 workspace 参数；has_pending_pipeline 按 workspace 过滤
state_machine_engine.pyStateMachineRuntime.__init__ 增加 workspace/pipeline_id 参数；_save_state/load_state 走管线子目录
handoff_ticket.py工单存储路径支持 workspace
cycle_bridge.py (MCP)get_cycle_state/queue_next_prompt 增加 workspace 参数

4.3 向后兼容

  • 旧 .reasonix/pipelines/pl-*.json → 自动迁移到 .reasonix/pipelines/.default/{id}/pipeline.json
  • 旧 .reasonix/state/machine_state.json → 迁移到默认管线子目录
  • 不指定 workspace 参数时，使用 .default workspace

6. 验收标准

  1. init_pipeline("test") 在 .reasonix/pipelines/.default/{id}/ 下创建完整文件
  2. init_pipeline("quant", workspace="quant") 在 .reasonix/pipelines/quant/{id}/ 下创建
  3. 两条管线可独立推进，状态不互相影响
  4. list_pipelines() 返回所有 workspace 的所有管线
  5. 旧 .reasonix/pipelines/pl-*.json 在首次读取时自动迁移
  6. 不指定 workspace 的 CLI 命令向后兼容

7. 不做什么

  • 不改管线流转逻辑本身（只改存储路径）
  • 不改 role-card/skill 注册机制
  • 不改 state-machine.yaml 定义格式
  • 不改 cycle-bridge MCP 协议的消息格式（只加参数）
