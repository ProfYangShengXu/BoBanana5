ADR-001：定时任务通过 MCP Tool 暴露

已接受 2026-07-16 v1

背景

Bobanana 5.0 已有 scripts/pipeline_scheduler.py 实现完整的 CLI 定时调度能力（schedule/run/list/remove/install-winsch），但 只能通过命令行操作。Reasonix 桌面端用户无法在聊天界面或 GUI 中管理定时任务。需要决定如何暴露调度能力给桌面端用户。

可选方案

方案描述优点缺点
A. MCP Tool 暴露（选定） 在 .mcp.json 中注册 schedule_* MCP tools，mcp_server.py 新增 handler，复用 pipeline_scheduler 现有逻辑 • 最小代码改动
• 向后兼容
• 用户在 Reasonix 聊天中即可操作
• 复用已有 schtasks 集成		 • 需新增 4-5 个 handler
• 同步阻塞问题
B. 独立 Web UI 用 Flask/FastAPI 启动一个本地 Web 服务提供调度管理界面 • 功能完整
• 可嵌入 iframe		 • 新增依赖和端口
• 偏离 Reasonix 桌面端体验
• 安全策略复杂
C. 仅 TUI 仪表盘 只在 gui_dashboard.py 中新增调度面板，不暴露 MCP tool • 最简实现 • Reasonix 桌面端用户无法操作
• 仅限终端用户使用
D. APScheduler 替换 schtasks 用 APScheduler 替代 Windows Task Scheduler 作为后端触发引擎 • 跨平台
• 更精确的控制		 • 大改现有架构
• 需进程常驻
• schtasks 对桌面工具更自然

决策

选定方案 A：MCP Tool 暴露。

理由：

  1. 最小改动原则：复用已有 pipeline_scheduler.py 的全部 CLI 逻辑，只需新增 MCP 胶水层
  2. Reasonix 桌面端原生集成：用户在聊天中通过 MCP tool 创建定时任务，零离开 IDE 体验
  3. 向后兼容：CLI 用户不受影响
  4. schtasks 对 Windows 桌面用户是最自然的系统级调度方案（即使用户关闭桌面端也能触发）

影响

  • 正面：用户可在 Reasonix 桌面端聊天中用自然语言创建定时任务（通过 AI 调用 MCP tool）
  • 正面：TUI 仪表盘也可通过 MCPClient 的 schedule_* 方法显示调度信息
  • 负面：Windows 独占（schtasks），但本版本目标平台就是 Windows Reasonix 桌面端
  • 负面：MCP handler 中调用 scheduler 是同步的（subprocess），但单次操作耗时 < 1s 可接受
  • 约束：所有 MCP handler 必须做严格的输入校验（命令注入防护）

状态

✅ 已接受 — 将在 v1 版本中实现
