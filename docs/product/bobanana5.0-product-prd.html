1. 背景与目标

现实问题

当前 AI 编程管线（包括 bobanana4.0 在内）都遵循一个本质假设：角色和流程是固定的。架构→开发→测试→评审，线性推进，状态机写死在 Go 代码里。这带来了几个根本局限：

  • 管线"硬编码"了领域知识：bobanana4.0 的 6 个角色（架构/开发/测试/评判/挑刺）绑定了"写代码"这个场景。换到游戏开发、数据分析、嵌入式等方向，同一个管线跑不通。
  • 状态机不可编程：角色转换逻辑写在 cycle-bridge 的 Go 代码中。想改一条转换规则就得改 Go → 编译 → 部署。非技术人员无法自定义工作流。
  • 角色与实现耦合：每个角色的行为逻辑和"角色定义"写在一起。没法在一个项目中定义"战斗策划"角色，在另一个项目中定义"数据科学家"角色，再统一接入同一个编排引擎。
  • 跨 session 交接有损：虽然用 [GOAL]/[STATE]/[NEXT] 模板传递上下文，但缺乏标准化的"接口契约"——接收方不知道上一个角色到底产出了什么、留下了什么未决决策。

一句话定位

Bobanana 5.0 — AI Agent 编排引擎。
领域无关的可编程状态机 + 可插拔角色卡系统。它不关心你写的是什么领域的代码、游戏还是文档——你只需要提供角色卡（定义每个角色做什么、输入输出）和状态机配置（定义角色间的流转规则），Bobanana 5.0 负责编排 Agent 在角色间流转、维护上下文、保证质量门。

Bobanana 5.0 本身不包含任何领域知识。领域知识（角色定义、工具调用、质检标准）全部由外部的 Adaption Kitsets 提供。核心编排层只做一件事：按状态机定义调度 Agent 角色，维护上下文一致性，确保质量门通过。

🧩 类比：Bobanana 5.0 核心 = 操作系统内核（调度+IPC+内存管理）；Adaption Kitsets = 驱动程序（显卡/网卡/硬盘各自独立）；外部工具 = 硬件设备。更换适配集 = 更换驱动，内核不用改。

现存方案痛点

  1. 角色与领域耦合：bobanana4.0 的 6 角色管线绑定了软件开发领域，无法直接用于游戏开发、数据分析等方向。
  2. 状态机硬编码：角色转换逻辑写在 Go 代码（cycle-bridge）中，修改需要重新编译，非技术人员无法自定义。
  3. 缺乏角色卡抽象：角色的行为、输入输出、质量门没有标准化的声明式定义格式。
  4. 跨 session 上下文丢失：虽然用 [STATE]/[NEXT] 模板传递，但缺乏结构化的"接口契约"。交接时经常信息丢失。
  5. "管线"是一等公民，但"角色网络"不是：当前系统假设 A→B→C 线性推进。真实协作是网络化的（并行、分支、反馈循环）。

1.2 三层架构详解

Layer 1: 核心编排层（Bobanana 5.0 本体）

这是一个领域无关的引擎，核心能力只有三条：

  • 状态机引擎 — 读取 YAML 定义的状态机配置（节点=角色、边=转换规则、条件=分支逻辑），驱动 Agent 在角色间流转。支持串行/并行/分支/循环。
  • 角色卡注册表 — 角色卡的"元数据"描述库。每张角色卡描述了一个角色能做什么、需要什么输入、产出什么、受什么质量门约束。编排层不执行角色卡逻辑，只负责在合适的时间把合适的 Agent 送入合适的角色。
  • 交接工单系统 — 标准化的上下文传递机制。角色 A 完成时生成工单（产出物列表 + 未决决策 + 风险标注），角色 B 启动时读取工单。工单是版本化的、可审计的。

Layer 2: Adaption Kitsets（外部项目）

每个适配集是一个独立的 Git 仓库，包含：

  • 角色卡文件 — 本领域特有的角色定义（如"战斗策划"、"技术美术"），YAML + Markdown 格式
  • 工具实现 — Agent 在该领域干活时调用的 CLI/MCP/脚本工具
  • 预设状态机模板 — 针对该领域推荐的管线拓扑（如游戏开发可能走 8 角色网络，Web 开发走 4 角色串行）
  • 质量门规则 — 领域特定的验收标准
  • 文档和模板 — 快速上手、最佳实践
规划中的例子：Godot-adaption-kitset — 设计为独立适配集仓库（当前路径不存在），相关 Skill 代码已在本代码库中
包含 4 个 Reasonix skill（godot-dev / godot-test / godot-debug / godot-docs）、11 个 TypeScript 工具模块、Godox Godot 插件、3 个游戏模板、4 层测试体系。完整实现了从"用户说一个游戏创意"到"可运行的 Godot 游戏"的 AI 全流程。现有代码需重构为标准角色卡格式放入独立适配集仓库。

1.3 角色标签体系

为了让管线从“固定流水线”升级为“网络化协作”，Bobanana 5.0 引入了一个关键抽象：角色标签。每张角色卡可以被打上一个或多个标签，标签决定了这个角色在管线中的“层级”——它是更改管线本身的（OP），还是终极审查者（CL），还是普通的技术执行角色。

OP Orchestration — 编排者标签

受用角色：架构师、技术经理、Project Lead、技术总监

行为：OP 角色不仅产出领域交付物（架构文档、任务拆解），还担当管线的“编排者”。它需要：

  • 读取当前项目中所有可用的角色卡及其简介（来自适配集 skills/roles/ 目录）
  • 选择一个 OP 或 CL 标签的角色卡作为出口（终审角色）
  • 选择若干个普通角色卡作为状态机节点（执行角色）
  • 动态编写或更新 state-machine.yaml，将入口→节点→出口连接成完整管线拓扑
  • 如遍历角色卡后发现缺少所需角色，触发紧急招聘：OP 在 state-machine.yaml 中插入 HR 作为临时节点，OP→HR→OP 流程由引擎自动编排，HR 完成后 OP 恢复编排

缺口检测规则：OP 遍历角色卡库时，若发现管线需要某类角色（如“需要一名数值策划”但库中没有对应角色卡），OP 不跳过也不妥协——而是触发紧急招聘，调用 HR 角色卡创建所需角色，待 HR 完成并注册新角色卡后，OP 继续编排。

核心理念：管线不是固定的——它是由 OP 角色根据项目实际情况“编程”出来的。不同项目阶段、不同产品类型可以有完全不同的管线结构。

CL Client Gate — 甲方终审标签

受用角色：用户/甲方代表、业务方审核员、产品总监

行为：CL 是整个管线的唯一终结出口。它在所有技术性角色（架构、开发、测试、评判、挑刺——即使它们中有 OP 标签）完成后执行，作为最后一道关卡。

  • 必须启动一个新的、完全隔离的 subagent（新的 agent 实例，不带任何 pipeline 历史上下文），以确保审查的客观性和公正性
  • 对整个项目代码进行严肃的全面审查，从 1 到 10 打分
  • 如果分数 < 9 分：将审查报告和分数一并打回给最近的 OP 角色（通常是架构师），由 OP 重新安排修复
  • 如果分数 ≥ 9 分：调用 signal_done，管线正式结束

核心理念：技术性评估（测试、评判、挑刺）和“业务方接受”是两回事。CL 角色睛主方视角看待整个产出物，而不只看技术合格不合格。

标签组合示例

一个角色可以同时拥有多个标签，也可以没有特殊标签。例如：

角色标签说明

HR 招聘专员HR仅由 OP 触发，紧急招聘新角色

受用角色：HR 招聘专员（一张特殊的角色卡，代表 HR Skill 本身）

约束：

  • HR 标签角色不被 OP 选作出口或普通节点——它是一张“服务角色卡”，只响应 OP 的紧急招聘请求
  • HR 不被用户或其他角色直接调用——唯一的入口是 OP 在遍历角色卡库发现缺口时触发的紧急招聘
  • HR 执行完毕后控制权回到 OP，OP 读取新注册的角色卡再继续编排

1.4 完整管线流程

结合 OP 和 CL 标签，一个完整的 Bobanana 5.0 管线流程可以表示为：

┌───────────────────────────────────────────────────┐ │ [OP] 架构师 │ │ • 读取项目目标 + 当前状态 │ │ • 产出技术 PRD + 任务拆解 │ │ • 动态编写 state-machine.yaml │ │ • 设置下一个 OP 角色或终审出口 │ └───────┬─────────────────────────────┘ │ │ 管线按 OP 定义的状态机执行 │ ┌────────┬───────────────────────────┐ │ 技术性角色链 │ │ 开发 → 测试(U/I/S/A) → 评判 → 挑刺 │ │ • 每个角色完成时 queue_next_prompt │ │ • 有 badcase 则 feedback 回开发 │ └───────┬────────────────────────────┘ │ │ 技术链全部 PASS │ ┌────────┬───────────────────────────────────┐ │ [CL] 甲方终审 │ │ • 新的隔离 subagent 审查 │ │ • 全项目代码打分 1-10 │ │ ────────────────────────── │ │ 分支： │ │ 分数 < 9 → 打回给 [OP] 架构师 │ │ 分数 ≥ 9 → signal_done ✔️ │ └────────────────────────────────────────┘

关键规则：

  • CL 是唯一出口。即使技术链全部 PASS，管线也不会自动结束——必须等 CL 审查通过。这保证了“技术合格”和“业务接受”之间的隔离。
  • OP 可以叠加。一个复杂项目可能有多层 OP：项目级架构师设置第一层状态机，子模块的技术经理再细化。但 CL 始终只有一个。
  • 打回链。CL 打回时不是打回给最后一个技术角色，而是打回给最近的 OP 角色，由 OP 判断是直接找开发修还是重新规划管线。

1.5 HR 招聘技能与角色卡创建

角色卡不是凭空产生的——它们需要被"招聘"出来。Bobanana 5.0 引入一个专门的 HR Skill（招聘技能），职责是在创建角色卡时对一个职业进行深度研究，产出标准化的角色卡文件并使其立即可在当前管线中使用。

HR Skill 招聘流程

当管线需要一个新的角色时，用户或 OP 角色调用 HR Skill，传入角色名称和简要描述。HR Skill 自动执行：

  • 第 1 步 派 4 个并行 subagent 做职业研究：
    • Subagent A 工作准则（Standards）：行业标准、最佳实践、常见规范
    • Subagent B 工作内容（Deliverables）：交付物类型、质量标准、产出格式
    • Subagent C 工具与方法（Tools & Methods）：常用软件、框架、工作流程
    • Subagent D 常见坑与风险（Pitfalls）：易犯错误、重工诱因、反模式
  • 第 2 步 合并研究结果为职业准则简报
  • 第 3 步 生成标准角色卡：输出 role-card.yaml + SKILL.md
  • 第 4 步 注册到当前管线：角色卡立即生效，OP 角色可在下一次状态机编排中引用它

HR Skill 与现有的准则文件

Bobanana 4.0 已有两个可视为预置职业准则文件的例子。HR Skill 可以跳过研究步骤直接引用它们：

例子 A：docs skill / 产品经理角色
定义了产品经理的工作方法：不写代码、只回答 WHAT/WHY、输出 HTML+YAML 双件。
例子 B：Bobanana.md / 架构师与开发角色
定义了 9 条架构原则、禁止面向结果编程、不重构没问题代码。

关键设计原则

  • 研究发生在招聘时，不发生在上班时——职业研究是一次性的成本，在创建角色卡时完成。角色卡创建后每次被唤起时直接使用卡中准则，不再重复研究。
  • 角色卡立即可用——写入 skills/roles/ 后，状态机引擎立刻能发现它。
  • 预置准则 = 省成本——适配集作者可预置准则文件，HR Skill 直接引用、不启动 subagent。
HR Skill 被调用 (角色名 + 简述) | 派出 4 个并行 subagent 做职业研究 +---------+---------+---------+---------+ | 工作准则 | 工作内容 | 工具方法 | 常见坑 | +---------+---------+---------+---------+ | 合并为职业准则简报 | 生成标准角色卡 (YAML + Markdown) | 写入 skills/roles/<角色名>/ | OP 角色在下轮读取可用角色卡 | 读简介 + 选出口(OP/CL) + 选节点 + 写状态机 v

HR 角色卡与紧急招聘流程

HR Skill 本身也是一张标准角色卡，拥有自己的 role-card.yaml 和 SKILL.md，可在角色卡注册表中被 OP 发现。但它与其他角色卡有本质不同：

紧急招聘流程：OP → HR → OP

  • 第 1 步 OP 遍历角色卡库发现缺口：架构师/技术经理读取所有角色卡后，发现某环节无对应角色。不跳过也不妥协。
  • 第 2 步 OP 触发紧急招聘：在 state-machine.yaml 中插入 HR 临时节点，传递角色名称和描述。
  • 第 3 步 HR 执行招聘：引擎流转 HR，派出 4 subagent 研究 → 生成角色卡 → 写入库。
  • 第 4 步 控制权回 OP：引擎流转回 OP，OP 重新读库（新卡已就位），继续编排。

约束：HR 角色卡不被任何角色直接调用。唯一入口是 OP 遍历缺口时触发的紧急招聘。

OP 遍历角色卡库 | 发现缺口 (缺少某类角色卡) | OP 在状态机中插入 HR 临时节点 | 引擎流转到 HR 角色 | HR 派出 4 subagent 做职业研究 | 生成标准角色卡 + 写入 skills/roles/ | 引擎流转回 OP | OP 重新读取角色卡库 (新卡已就位) | OP 继续编排完整状态机 v

4. MVP 定义

Must-have（按优先级排序）

  1. 角色卡系统（Role Card Registry） — 标准化角色卡片格式（YAML Schema）。每张卡片含：角色 ID、名称、描述、输入契约、输出契约、适用阶段、标签。角色卡存储在 skills/roles/<角色名>/，编排层只读元数据、不执行业务逻辑。
  2. 可编程状态机（Programmable State Machine） — 状态机定义从 Go 代码移到 YAML 配置文件 state-machine.yaml。支持：节点（角色）、边（phase flag + 下一角色）、条件分支（判断逻辑）、并行 fork/join、循环反馈。适配集自带预设状态机模板。
  3. 交接工单系统（Handoff Ticket） — 结构化上下文传递，替代纯文本 [STATE] 模板。工单含：发送者 ID、接收者 ID、产出物列表（带路径）、未决决策（列表）、显式假设、风险标注。工单是版本化的 JSON 文件，存储在 .reasonix/handoffs/。
  4. 角色卡验证器 — 在 install.bat 或 reasonix cycle 启动时自动校验角色卡 YAML 格式。不符合 Schema 的卡给出具体错误位置和修复建议。
  5. 状态机图形化输出 — 每轮角色转换后输出 Mermaid 格式的状态机图，嵌入 HTML 或控制台。用户一眼看清当前管线状态（已过角色 / 当前角色 / 待过角色 / 失败计数）。
  6. 向后兼容 — bobanana4.0 的 /pipeline、/loop、/cycle 命令继续可用。预设的"通用软件开发"状态机模板覆盖 4.0 的全部行为。
  7. 首个官方适配集：Godot-adaption-kitset — 将现有的 Godot-adaption-kitset 重构为角色卡格式，作为参考实现和验证案例。

角色标签特定功能（must-have）

  1. OP 编排者标签 — 角色卡 Schema 支持 tags: [op]。OP 角色运行时有权读取和写入 state-machine.yaml，可动态添加/删除/修改角色节点、转换边和分支逻辑。引擎在 OP 角色 session 结束后重新加载状态机配置。
  2. CL 甲方终审标签 — 角色卡 Schema 支持 tags: [cl]。CL 角色定义为整个管线的唯一出口：当管线到达 CL 时，引擎启动一个全新隔离的 subagent（不继承任何 pipeline 对话历史、不共享上下文），对整个项目代码执行全量审查。
  3. CL 打分 + 打回机制 — CL 审查报告强制包含 1-10 整数评分。评分 < 9 时引擎自动将审查报告 + 分数通过 handoff ticket 送回给最近的一个 OP 角色（通常是架构师），由 OP 决定修复方案。此前已完成的技术角色产出物保留、不清除。评分 ≥ 9 时触发 signal_done。

HR 招聘技能（must-have）

  1. HR Skill 角色 — 一个专门的 Reasonix skill，负责职业研究 + 角色卡创建。在需要新角色时被调用。
  2. 4 子 agent 并行研究 — HR Skill 自动派出 4 个并行 read-only subagent，分别研究工作准则、工作内容、工具与方法、常见坑与风险，合并为职业准则简报。
  3. 标准角色卡生成 — HR Skill 产出标准化的 role-card.yaml + SKILL.md，写入 skills/roles/，立即可被管线发现和引用。
  4. 可引用已有准则文件 — 如果适配集已预置职业准则文件（docs skill / Bobanana.md），HR Skill 直接引用、跳过 subagent 研究。
  5. 一次研究、多次使用 — 职业研究在创建角色卡时一次性完成。角色卡创建后每次被唤起时直接使用卡中准则，不再重复研究。

Nice-to-have

  • 角色卡市场 / 仓库索引（从远程拉取适配集）
  • GUI 状态机编辑器（拖拽连线）
  • 角色卡版本管理 + 适配集依赖声明
  • 管道并行：同一时刻多个 Agent 运行不同角色
  • 适配集评分 / 质量评级体系

明确不比

  • 不比领域工具：不内置任何领域的 SDK、编辑器插件、渲染引擎。Bobanana 5.0 不写代码、不画图、不渲染——它只负责调度。干活的是 Adaption Kitset 里的工具。
  • 不比项目管理工具：不跟 JIRA、Notion 争甘特图和看板。它是 AI 执行引擎，不是管理面板。
  • 不比模型训练平台：不训练 AI 模型。使用 Reasonix 对接的 LLM API。
  • 不比无代码平台：不是给非技术人员"拖拽做应用"的。它是给 AI Agent（和懂技术的人）用的编排层。

7. 设计理念

体验关键词

  • 🧩 模块化 — 角色是积木，状态机是搭建说明书。每一块独立、可替换、可组合。核心不关心积木是什么颜色。
  • 🔌 可插拔 — 适配集=驱动。换适配集=换领域。核心引擎不改一行代码。驱动出了问题换驱动，不重装操作系统。
  • 📐 声明式 — 状态机配置、角色卡定义、交接工单全部用 YAML 声明式描述。用户描述"是什么"，不描述"怎么做"。
  • 🔍 可审计 — 每一次角色转换、每一张工单、每一个质量门检查都有记录。追溯问题和归因不需要翻对话历史。

核心交互原则

  1. 一次一个角色 — 每个 session 只扮演一个角色，产出该角色的交付物。角色完成时自动触发交接，不征求同意（RED LINE 继承）。
  2. 交接即工单 — 角色间不走"对话历史"，走结构化工单。工单是版本化的 JSON，存储在 .reasonix/handoffs/，可审计、可回溯。
  3. 状态机看得见 — 每轮输出 Mermaid 状态机进度图 + 统计（当前阶段 / 已过 / 待过 / 重试次数），用户始终知道"现在谁在干活"。

视觉风格方向

产品文档和角色卡使用清爽的卡片式 UI，核心编排层用深紫/深蓝色系，适配集用不同的主题色区分。用户界面保持"信息密度高、功能优先"的开发者工具风格。所有输出（状态机图、工单视图、质量门报告）都支持 HTML 和 Markdown 两种格式。
