🏢 Bobanana 5.0 — Adaption Kitset 接入规范

版本: v1.0 | 日期: 2026-07-14 | 类比: 子公司与母公司

1. 核心理念：子公司模式

类比：Bobanana 核心 = 母公司（集团总部），Kitset = 子公司（专业子公司）。
母公司提供：品牌(Bobanana)、基础设施(状态机引擎)、公共服务(HR/CL)、渠道(Boss入口)。
子公司提供：领域专业能力(角色卡)、专用工具(工具集)、领域OP管理(嵌入式架构师)。
铁律：子公司不改母公司代码，除非改动对所有子公司都有利。
角色母公司(Bobanana核心)子公司(Kitset)
入口Boss（发现+匹配+切换）嵌入式架构师（编排+执行+终验）
角色池59张通用角色卡17张嵌入式专用角色卡
工具通用工具cubemx/keil5/jlink等嵌入式工具
状态机通用模板(v15 standard)state-machine-embedded.yaml
CL统一CL终审（隔离审查）使用母公司CL
HR统一HR招聘可使用母公司HR招聘到子公司

2. 发现阶段（Boss启动时）

Boss 启动时自动扫描 kitsets/ 目录下所有适配集。

kitsets/
├── embedded/           ← 嵌入式系统开发 Kitset
│   ├── reasonix-plugin.json   ← 元数据（kitset_name, kitset_domains）
│   ├── roles/                 ← 领域角色卡
│   ├── tools/                 ← 领域工具集
│   └── state-machine-embedded.yaml
└── ... (更多Kitset)

每个 Kitset 通过 reasonix-plugin.json 声明自己的领域标签：

{
  "kitset_name": "embedded",
  "kitset_domains": ["embedded", "firmware", "stm32", "fpga",
                     "linux-embedded", "hardware", "rtos"],
  "entry_role": "embedded-architect"
}

3. 匹配阶段（Boss分析目标时）

Boss 分析用户目标，按关键词匹配合适的 Kitset：

用户目标关键词匹配Kitset操作
"STM32"、"固件"、"嵌入式"、"MCU"、"RTOS"、"电机"embedded切换控制权
"FPGA"、"Verilog"、"VHDL"、"逻辑分析"embedded切换控制权
"Linux驱动"、"BSP"、"Yocto"、"Buildroot"embedded切换控制权
其他通用目标无匹配使用默认角色池
匹配原则：匹配置信度≥2个标签自动切换，1个标签询问用户，0个标签用默认池。

4. 切换阶段（Boss移交控制权）

匹配成功时，Boss 执行控制权移交：

1. Boss 检测到用户目标匹配 embedded kitset
2. Boss 读取 state-machine-embedded.yaml
   → entry_point: "embedded-architect" (kitset的OP角色)
3. Boss 写入 state-machine.yaml
   → 将entry_point改为"embedded-architect"
   → 或插入边: "boss → embedded-architect"
4. Boss调用 queue_next_prompt(phase="boss-done")
5. 下一Session: embedded-architect 登场
→ 控制权从 Boss(母公司) → embedded-architect(子公司)

5. 执行阶段（Kitset OP编排）

嵌入式架构师(OP)使用母公司的状态机引擎编排 Kitset 内角色：

embedded-architect
  ├── 读取 state-machine-embedded.yaml
  ├── 调用 state_machine_engine.load() [母公司引擎]
  ├── 遍历角色链: 固件→Linux→应用→FPGA→硬件→安全
  ├── 各角色调用 queue_next_prompt 推进
  ├── 可选: 调用母公司HR招聘新角色进Kitset
  └── 完成: 调用 queue_next_prompt(phase="arch-gate-pass")

关键：Kitset OP 与通用架构师的职责相同（读需求→拆任务→编排→终验），但使用领域专用的角色卡和工具集。

6. 退出阶段（CL终审）

Kitset 执行完成后回到母公司 CL 终审：

Kitset内角色链全部完成
  → embedded-architect 终验
  → client-gate (母公司CL，隔离审查)
     ├── score ≥ 9 → __terminal__ ✅
     └── score < 9 → 打回 embedded-architect 🔄

7. 铁律

Kitset 不得修改以下母公司代码：
state_machine_engine.py / pipeline_orchestrator.py / role_tag_system.py
role_card_registry.py / mcp_server.py / Bobanana.md / reasonix.toml

Kitset 可以操作的内容：
kitsets/<name>/ 下全部文件 / state-machine.yaml（运行时）/ 管线过程中 docs/

8. 一句话总结

Kitset = 子公司的角色卡+工具集，挂载在母公司 Bobanana 的状态机引擎上，Boss自动发现并按需切换OP角色，不改一行母公司代码。
