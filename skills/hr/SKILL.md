# hr — HR 招聘技能

**语言指令：推理用英文，回复用中文。**

## 核心原则

```
本 skill 只回答 WHO 和 WHAT，不回答 HOW。
生成的角色卡必须对齐 pipeline 角色格式（使命/步骤/原则/质量门/RED LINE/完成框）。
必须通过 Schema 验证，否则重试。
```

## 产出物

每张角色卡产出 3 个文件，存放在 `skills/roles/<角色名>/` 目录下：

1. `role-card.yaml` — 结构化角色定义（pipeline 架构师直接引用）
2. `SKILL.md` — **对齐 pipeline 格式的可执行角色定义**（包含完整的使命、步骤、铁律、完成框）
3. `standards-brief.yaml` — 职业准则简报

三个文件必须同时产出，内容一致。

## 招聘流程（4 步）

### 第 1 步：派 4 个并行 subagent 做职业研究

| Subagent | 研究方向 | 具体内容 |
|----------|---------|---------|
| **A — 工作准则** | 行业标准、最佳实践 | 该职业的行业标准、核心方法论、常见规范、参考框架 |
| **B — 工作内容** | 交付物类型、质量标准 | 该职业每天做什么、产出什么、质量标准是什么、审查流程 |
| **C — 工具与方法** | 常用软件、框架、工作流程 | 用什么工具、什么工作流、如何协作 |
| **D — 常见坑与风险** | 易犯错误、重工诱因、反模式 | 最常犯什么错、什么决策导致重工、团队有什么反模式 |

4 个 subagent **并行执行**，互不依赖。
每个 subagent 输出一份结构化的研究笔记。

### 第 2 步：合并为职业准则简报

合并 4 份研究笔记，产出 `standards-brief.yaml`。根据角色类型（开发/测试/架构/设计/运营/管理），选择对应的 SKILL.md template。

### 第 3 步：生成标准角色卡

根据上面合并的简报，生成 `role-card.yaml`（含输入/输出契约 + 质量门）。
同时按照角色类型 template 生成 `SKILL.md`（对齐 pipeline 格式，含使命/步骤/原则/RED LINE/完成框）。

### 第 4 步：注册到管线

1. 运行 `validate_role_card.py` 验证通过
2. 运行 `role_card_registry.py scan` 注册到管线
3. 告知用户 `/hr recruit <角色名> "<描述>"` 可再次调用

## 快速引用模式

如果该职业已有现成的准则文件（如 `Bobanana.md` 之于开发、`skills/docs/SKILL.md` 之于产品经理），可以跳过第 1 步和第 2 步，直接引用：

```bash
python hr_recruitment.py quick "<角色名>" "<准则文件路径>"
```

这种方式只产出角色卡，不派 subagent，节省 API 成本和耗时。

## 约束

- 角色名必须用英文 kebab-case
- SKILL.md 必须包含：使命、步骤、核心原则、质量门、不做、RED LINE #7、完成框
- role-card.yaml 必须通过 Schema 验证
- 必须同时产出 role-card.yaml + SKILL.md + standards-brief.yaml

## 不做什么

- 不修改已有角色卡（覆盖前需确认）
- 不生成带 OP/CL/HR 标签的角色（标签由人工添加）
- 不部署、不运行、不测试角色——只创建定义
- 不写实现代码、不改已有代码
