1. 用户访谈提纲 🎙️

目标用户画像

  • 主用户：Reasonix 桌面端日常使用者，用 Bobanana 管线做持续开发/自动化
  • 次用户：需要定时运行 AI 管线的开发团队负责人、DevOps 工程师
  • 反用户：仅一次性手动使用管线、不需要定时运行的轻量用户

招募条件

  • 使用过 Reasonix 桌面端 ≥1 周
  • 有过"希望让 AI 在后台跑任务"的想法
  • 了解 CLI 基本操作

访谈问题

Q1. 你目前如何安排需要定时运行的开发任务？（如定时跑测试、定时拉数据）
Q2. 你在 Reasonix 桌面端最常做的三件事情是什么？
Q3. 如果能在桌面端直接创建一个定时任务（如"每天早9点跑一次数据分析管线"），你会用吗？
Q4. 你希望定时任务的粒度是？（每日/每周/每小时/自定义 cron）
Q5. 你希望定时任务执行完毕后如何通知你？（桌面通知/日志/邮件）
Q6. 你会介意任务执行时 Reasonix 桌面端需要保持打开状态吗？
Q7. 你觉得创建一个定时任务需要多少步才叫"简单"？
Q8. 你用过其他 AI 工具的定时任务功能吗（Claude Code Routines / Cursor Automations）？体验如何？

2. 价值主张画布 💎

用户任务 (Customer Jobs)

功能型：定时运行 AI 管线（测试/代码审查/报告生成）
社交型：向团队成员展示"管线已自动化"
情感型：从手动触发→自动运行的"省心"感

痛点 (Pains)

P1 每次都要手动打开终端输入命令 → 繁琐
P2 记不住/忘记在特定时间运行管线
P3 Windows 任务计划程序配置复杂，不熟悉的用户不会用
P4 现有 CLI 调度器无 GUI，看不到任务状态
P5 无法在 Reasonix 桌面端内完成全流程

收益 (Gains)

G1 在 Reasonix 桌面端一站式完成创建→管理→监控
G2 无需离开 IDE 即可管理所有定时任务
G3 任务执行历史一目了然
G4 可与已有管线无缝衔接

产品方案对应

痛点/收益产品方案
P1 手动输入命令MCP Tool 直接创建，用户在 Reasonix 聊天中自然语言完成
P2 忘记定时执行注册后自动由 schtasks/APScheduler 触发
P3 配置复杂自然语言描述 → 自动解析为调度配置
P4 看不到状态TUI 仪表盘 + MCP tool 查询实时状态
P5 不能全流程从创建到日志到删除全部在桌面端完成

成功标准：≥60% 用户首次尝试成功创建

7. 一句话定位卡片 🎯

方案 A — "AI 管线的智能闹钟"
目标用户：Reasonix 桌面端的日常用户
解决什么：不想手动触发定时管线
为什么选你：在 Reasonix 桌面端内自然语言创建，无需离开 IDE

8. 需求评审清单 ✅

#自查问题为什么重要
1用户创建定时任务需要几步？门槛越低，使用率越高。目标：1 句自然语言 ≤ 2 步确认
2出错时用户知道如何解决吗？错误信息必须可理解，而不是 Python traceback
3已有 CLI 功能是否会中断？向后兼容是硬约束
4定时任务与状态机管线如何关联？执行的是完整管线而非孤立脚本
5定时任务执行时桌面端是否需要打开？依赖 Windows Task Scheduler，不需要
6多个定时任务同时触发会怎样？需要排队/互斥机制，不能冲突
7MCP tools 的权限是否足够？需要 write 权限创建任务，read 权限查询
8是否会增加安全风险？操作范围限定在 pipeline 调度，不暴露任意命令执行

9. 一句话总结 📝

一句话总结：在 Reasonix 桌面端内通过 MCP tool + 自然语言创建定时任务，复用现有 `pipeline_scheduler.py` 的 Windows Task Scheduler 后端，让用户像设闹钟一样为 AI 管线安排定时执行，零 CLI 操作。
