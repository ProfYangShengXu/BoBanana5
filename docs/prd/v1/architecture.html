4. 核心逻辑

4.1 WorkspaceManager 辅助类

class WorkspaceManager:
    """工作区路径管理器"""

    BASE = ".reasonix/pipelines"

    @staticmethod
    def get_workspace_dir(workspace=None):
        ws = workspace or get_default_workspace() or ".default"
        return os.path.join(WorkspaceManager.BASE, ws)

    @staticmethod
    def get_pipeline_dir(pipeline_id, workspace=None):
        return os.path.join(
            WorkspaceManager.get_workspace_dir(workspace),
            pipeline_id
        )

    @staticmethod
    def get_default_workspace():
        """从项目配置或工作目录名推导默认 workspace"""
        return None  # 返回 None 时使用 ".default"

    @staticmethod
    def list_workspaces():
        """列出所有工作区"""
        base = WorkspaceManager.BASE
        if not os.path.isdir(base):
            return []
        return sorted([
            d for d in os.listdir(base)
            if os.path.isdir(os.path.join(base, d))
        ])

    @staticmethod
    def migrate_legacy():
        """将旧 .reasonix/pipelines/pl-*.json 迁移到新结构"""
        legacy_dir = os.path.join(".reasonix", "pipelines")
        if not os.path.isdir(legacy_dir):
            return
        for f in os.listdir(legacy_dir):
            if f.endswith('.json') and f.startswith('pl-'):
                src = os.path.join(legacy_dir, f)
                pid = f.replace('.json', '')
                dst_dir = os.path.join(legacy_dir, ".default", pid)
                os.makedirs(dst_dir, exist_ok=True)
                shutil.copy2(src, os.path.join(dst_dir, "pipeline.json"))
                os.remove(src)  # 删除旧文件

4.2 管线 ID 变更

不再使用 pl-{timestamp}.json 作为文件名。管线 ID 改为目录名 pl-{timestamp}，管线文件统一命名 pipeline.json。

4.3 向后兼容策略

  1. 首次调用 get_pipeline() 时，检查旧目录是否有 pl-*.json，如有则自动迁移到 .default/{pid}/pipeline.json
  2. 没有 workspace 参数时，所有路径自动回退到 .default 工作区
  3. 旧 machine_state.json：迁移到默认管线目录下
  4. 旧 cycle_state.json：如果管线 ID 可推断，迁移到对应管线目录

5. 风险评估

🔴 H1 — 迁移中断：旧管线在迁移过程中如果崩溃，新旧文件可能各写一半。
方案：迁移采用 copy-then-remove 原子模式。旧 pipeline.json 在 copy 成功前不删除。状态机引擎写 state 时使用临时文件 + rename。
🟡 M1 — MCP 不兼容：cycle-bridge MCP 工具不知道 workspace 参数，传进来的 phase 可能指向错误的状态文件。
方案：MCP server 端加 workspace 参数（可选，默认 .default）。不加参数的旧调用行为不变。
🟡 M2 — 多管线推进冲突：两条管线同时调用 advance_pipeline，检查 has_pending_pipeline 时互相误判。
方案：has_pending_pipeline(workspace="quant") 只检查 quant 下的未完成管线，互不干扰。
🟢 L1 — 目录膨胀：大量管线的子目录堆积。
方案：已完成管线自动压缩为 .archive.tar.gz，或达到 N 条时自动清理 30 天前的已结束管线。
